# IDOR

IDOR - (Insecure Direct Object Reference) уязвимость ненадлежащей проверки прав доступа, 
и доступом к данным, а сам Object Reference здесь, как идентификатор ссылки на запись в бд

### horizontal vs vertical privilege escalation

При эксплуатации этой уязвимости есть разделение повышения привелегий гооризонтальное и вертикальное 

Первое - это наиболее классический сценарий для IDOR, когда злоумышленник получает доступ к другому пользователю 
с такими же правами `GET /api/v1/get_profile?user_id=123` - при этом варианте возможно просто изменить 123 на 
124 и перейти на аккаунт на пользователя с id 124. Ранее, в другом отчете я обращал внимание на сессию куков, 
при расследовании запросов в Burp Suite, там первая часть представляла собой id пользователя, зашифрованный в Base64 
и код кук-сессии, идентичный пример, только здесь нужно было вручную рассматривать трафик

vertical заключается в том, когда обычный пользователь просматривает или изменяет данные администратора, пользуется его привелегиями,
в пример этого можно привести такой метод: `GET /api/v1/admin/export_logs?report_id=1`, здесь можно заметить, 
 специфический формат id и другой путь, при таком экспуатировании сервер может проверять только факт авторизации, а не 
привелегии, связанные с аккаунтом, с которого запрашивается метод

Возникает закономерный, вполне очевидный вопрос: "почему изменение id не равно автоматической уязвимости?", 
ответить на него очень просто, ранее, в отчете, в котором была описана попытка изменения зашифрованного session-cookie
код сессии кука не совпадал, что защищало аккаунт от подобной уязвимости, в других примерах сервер может просто грамотно
следить за правами пользователей, что даст разграничение возможностей аккаунтов и предотвратит подобные запросы
