# Crypto Company AWS Security Baseline — Sanitized

> **Sanitized version for portfolio.** Документ обезличен: название компании, внутренние аккаунты, архитектурные детали, поставщики и конкретные идентификаторы не раскрываются. Материал показывает формат работы с базовыми cloud security controls для небольшой крипто-компании.

## 1. Контекст

В рамках работы был подготовлен краткий security baseline для AWS-инфраструктуры небольшой крипто-компании. Основной фокус: доступы, аудит, мониторинг, резервное копирование, шифрование, сегментация сети, восстановление после инцидентов и управление рисками поставщиков.

## 2. IAM и управление доступом

Рекомендации:

- Включить MFA для root-аккаунта, администраторов, VPN и административных панелей.
- Использовать MFA на всех пользовательских аккаунтах, где это возможно.
- Не использовать root-аккаунт в повседневной работе.
- Убедиться, что root access keys отсутствуют.
- Исключить общие учётные записи.
- Применять принцип минимальных привилегий.
- Проводить регулярный review прав доступа.
- Хранить пароли и данные аккаунтов в защищённом хранилище с контролем доступа и MFA.

## 3. Аудит и мониторинг

### CloudTrail

- Включить аудит действий пользователей и сервисов.
- Обеспечить защищённое хранение логов.
- Ограничить доступ к логам.
- Настроить контроль изменений критичных настроек.

### CloudWatch

- Настроить мониторинг инфраструктуры.
- Добавить алерты по критичным событиям.
- Отслеживать аномалии в доступах, сетевой активности и работе сервисов.

## 4. RDS и хранение данных

Рекомендации:

- Включить резервное копирование.
- Ограничить сетевой доступ к базе данных.
- Использовать Multi-AZ для повышения отказоустойчивости.
- Включить Point-In-Time Recovery, если это поддерживается текущей архитектурой.
- Включить шифрование БД.
- Проверить шифрование дисков и object storage buckets.

## 5. Сетевая безопасность

- Использовать приватные подсети для внутренних сервисов.
- Разделять компоненты по подсетям и security groups.
- Ограничивать доступ к административным интерфейсам.
- Вести аудит VPN-подключений.
- Минимизировать публичную поверхность атаки.
- Проверить, что критичные сервисы не доступны напрямую из интернета без необходимости.

## 6. Защита клиентских и операционных данных

- Обеспечить надёжное хранение персональных и клиентских данных.
- Ограничить доступ к данным по ролям.
- Использовать шифрование для баз данных, дисков и object storage.
- Логировать доступ к критичным данным.
- Проверить процессы удаления, восстановления и резервного копирования данных.

## 7. Incident Management

Необходимо иметь базовый процесс реагирования на инциденты:

- кто отвечает за инцидент;
- как фиксируется событие;
- как проводится первичная оценка риска;
- как ограничивается ущерб;
- как восстанавливается работа;
- как документируются выводы после инцидента.

Примеры вопросов для сценариев отказа:

- Что делать, если критичный внешний поставщик временно недоступен?
- Что делать, если основная база данных недоступна?
- Что делать, если обнаружен подозрительный вход в административную панель?
- Кто принимает решение об отключении доступа или переводе сервиса в аварийный режим?

## 8. Vendor Risk Management

Для крипто-компании особенно важно учитывать риски внешних поставщиков:

- критичные API и инфраструктурные провайдеры;
- платёжные и custody-related сервисы;
- облачные сервисы;
- поставщики мониторинга, логирования и уведомлений.

Рекомендации:

- вести список критичных поставщиков;
- понимать, какие бизнес-процессы зависят от каждого поставщика;
- иметь контактные и escalation-процедуры;
- оценивать последствия недоступности поставщика;
- предусмотреть fallback-сценарии.

## 9. Business Continuity Plan

Для повышения устойчивости необходимо подготовить базовый план непрерывности:

- сценарии отказа критичных сервисов;
- порядок восстановления;
- ответственные лица;
- резервные каналы связи;
- RTO/RPO для ключевых компонентов;
- порядок проверки backup и recovery-процедур.

## 10. DORA-related направления

Для соответствия базовой логике операционной устойчивости стоит отдельно развивать:

- Incident Management;
- Vendor Risk Management;
- Business Continuity Plan;
- логирование и мониторинг;
- возможность восстановления после отказов;
- документирование критичных зависимостей.

## 11. Итог

Главный приоритет — усилить базовый cloud security posture: MFA, least privilege, защищённый аудит, мониторинг, резервное копирование, шифрование, приватные подсети, управление инцидентами и рисками поставщиков.

Документ не является полным аудитом AWS-инфраструктуры. Это краткий baseline и список направлений для дальнейшей проверки и внедрения.
